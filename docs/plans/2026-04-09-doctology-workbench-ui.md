# DocTology Workbench UI Implementation Plan

> For Hermes: Use subagent-driven-development skill to implement this plan task-by-task.

Goal: Build a workbench-first DocTology UI that makes pipeline health, page reading, ask/explore, and graph drill-down usable for real operators without turning the product into a graph toy.

Architecture: Keep the current DocTology stack as the backend source of truth. Add a thin UI shell that consumes stable summary/report APIs from canonical outputs and graph projection artifacts. The home screen is an operator dashboard, the default reading surface is a page/article view, and the graph is a bounded neighborhood inspector only.

Tech Stack: React + TypeScript + Vite, Tailwind CSS, TanStack Query, React Router, Zustand (small local UI state), FastAPI or lightweight Python JSON endpoints for summary APIs, WebGL-capable graph renderer (prefer Sigma.js-class path or other WebGL-first renderer), existing DocTology Python scripts and projection outputs.

---

## Non-negotiable product rules

1. `warehouse/jsonl/` remains canonical truth.
2. Graph projection remains derived and read-only.
3. The graph is never the home screen.
4. Never render the whole graph by default.
5. All graph views start from a seed node or answer context.
6. Page reading and provenance stay first-class.
7. Dashboard must reveal health, freshness, drift, and graph usefulness.

## API contracts to expose

Implement these read-only contracts first. They can start as JSON files generated by scripts, then grow into live endpoints.

### 1. Dashboard summary

Path:
- generated file: `artifacts/ui/dashboard_summary.json`
- optional API: `/api/dashboard/summary`

Fields:
- corpus freshness timestamps
- latest ingest run status
- canonical counts: documents, entities, claims, evidence, segments, derived_edges
- accepted claim ratio
- graph projection freshness
- pending review count
- recent page updates
- recent analyses
- warnings
- suggested next actions

### 2. Page index

Path:
- generated file: `artifacts/ui/page_index.json`
- optional API: `/api/pages/index`

Fields:
- page id
- title
- type
- summary
- aliases
- updated_at
- related ids
- source links

### 3. Page detail

Path:
- optional API: `/api/pages/:id`

Fields:
- title
- type
- summary
- sections
- claims
- evidence bundle
- related pages
- related entities
- provenance links
- graph seed id if available

### 4. Graph neighborhood

Path:
- generated file or API: `/api/graph/neighborhood?seed=<id>&hops=1|2`

Fields:
- seed node
- nodes
- edges
- edge family counts
- visible caps
- truncation flag
- baseline-vs-graph note

### 5. Doctor summary

Path:
- generated file: `artifacts/ui/doctor_summary.json`
- optional API: `/api/doctor/summary`

Fields:
- validator status
- stale projection warning
- docs/runtime drift warnings
- missing outputs
- graph regression warnings
- recommended repair commands

## Final design stance

This plan intentionally avoids building a graph spectacle.
It builds a trustworthy workbench first, then adds graph power where graph actually earns its keep.

The product should make users think:
- I know the system state
- I can ask useful questions
- I can read the answer in a stable page form
- I can inspect why things connect
- I trust the boundaries between truth, synthesis, and graph projection
